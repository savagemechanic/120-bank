# 120 Bank

120 Bank stores and reviews short-form social video ideas generated by Codex automations. SEIGHA's automation submits ideas. CHIBUEZE's agent reviews pending ideas. Humans approve, flag, discard, and copy approved briefs for production.

## License

MIT

## Stack

- Next.js App Router + TypeScript
- Route Handlers for API endpoints
- Postgres as the primary structured database
- Tailwind dashboard UI
- Vitest coverage for validation, idempotency, permissions, transitions, and dashboard smoke rendering

## Setup

```bash
cp .env.example .env
npm install
docker compose up -d
npm run db:migrate
npm run db:seed
npm run dev
```

Open the dashboard with a human review token:

```text
http://localhost:3000/?token=<HUMAN_REVIEW_TOKEN>
```

## Environment

```bash
DATABASE_URL="postgresql://postgres:postgres@localhost:5438/one_twenty_bank"
SEIGHA_AGENT_API_KEY="long-random-seigha-key"
CHIBUEZE_AGENT_API_KEY="long-random-chibueze-key"
HUMAN_REVIEW_TOKEN="long-random-human-token"
```

## Database

Migrations live in `migrations/`.

```bash
npm run db:migrate
```

The schema creates:

- `ideas`
- `idea_events`
- `idea_status` enum
- indexes for status, owner, cursor pagination, and audit lookup

## API Auth

Use either:

```bash
Authorization: Bearer <token>
```

or:

```bash
x-api-key: <agent-key>
```

Human review requests use:

```bash
x-review-token: <HUMAN_REVIEW_TOKEN>
```

All mutations require a valid server-side credential.

## SEIGHA Agent Examples

SEIGHA can only submit new ideas. The API always stores `owner = "SEIGHA"` and `status = "pending_review"` regardless of submitted owner/status fields.

```bash
curl -X POST http://localhost:3000/api/ideas \
  -H "Authorization: Bearer $SEIGHA_AGENT_API_KEY" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: seigha-idea-2026-06-24-001" \
  -d '{
    "submittedBy": "SEIGHA_CODEX_AUTOMATION",
    "title": "The Receipt Test",
    "concept": "A creator flashes three receipts and reveals the spending habit nobody admits before payday.",
    "storyboardBeats": [
      "Open with three receipts slapped on a table.",
      "Zoom into the smallest purchase that triggered the biggest regret.",
      "Close with a practical 10-second rule."
    ],
    "psychology": "Pattern interrupt, confession, then relief through a simple rule.",
    "whyItCouldPerform": "It gives viewers a familiar money shame moment and a save-worthy takeaway.",
    "caption": "One tiny receipt can expose the whole month.",
    "onScreenText": ["Which receipt hurt most?", "The 10-second rule"],
    "risks": ["Financial advice must stay general."],
    "sources": ["SEIGHA automation run"]
  }'
```

Repeat the same `Idempotency-Key` to safely retry without creating duplicates.

## CHIBUEZE Agent Examples

CHIBUEZE can read pending ideas only.

```bash
curl "http://localhost:3000/api/ideas?status=pending_review&limit=20" \
  -H "Authorization: Bearer $CHIBUEZE_AGENT_API_KEY"
```

Mark a pending idea reviewed:

```bash
curl -X PATCH "http://localhost:3000/api/ideas/$IDEA_ID/review" \
  -H "Authorization: Bearer $CHIBUEZE_AGENT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"note":"Ready for human approval."}'
```

Flag a pending idea:

```bash
curl -X PATCH "http://localhost:3000/api/ideas/$IDEA_ID/flag" \
  -H "Authorization: Bearer $CHIBUEZE_AGENT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"note":"Risk needs human review."}'
```

## Human Review Examples

Approve a reviewed idea:

```bash
curl -X PATCH "http://localhost:3000/api/ideas/$IDEA_ID/approve" \
  -H "x-review-token: $HUMAN_REVIEW_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"note":"Approved for production."}'
```

Flag an idea:

```bash
curl -X PATCH "http://localhost:3000/api/ideas/$IDEA_ID/flag" \
  -H "x-review-token: $HUMAN_REVIEW_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"note":"Needs source cleanup."}'
```

Discard an idea:

```bash
curl -X PATCH "http://localhost:3000/api/ideas/$IDEA_ID/discard" \
  -H "x-review-token: $HUMAN_REVIEW_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"note":"Not useful for production."}'
```

## Status Rules

- New SEIGHA submissions start as `pending_review`.
- CHIBUEZE can move `pending_review` to `reviewed`.
- CHIBUEZE can move `pending_review` to `flagged`.
- Humans can move `reviewed` to `approved`.
- Humans can flag active ideas.
- Humans can discard pending, reviewed, approved, or flagged ideas.

Every create and status change writes an `idea_events` audit row.

## Deployment

1. Create a Postgres database with Neon, Supabase, Vercel Postgres, or another managed Postgres provider.
2. Set `DATABASE_URL`, `SEIGHA_AGENT_API_KEY`, `CHIBUEZE_AGENT_API_KEY`, and `HUMAN_REVIEW_TOKEN` in the deployment environment.
3. Run migrations against the production database:

   ```bash
   DATABASE_URL="<production-url>" npm run db:migrate
   ```

4. Deploy to Vercel or any Node-compatible host:

   ```bash
   npm run build
   npm run start
   ```

5. Share the secret human review link only with reviewers:

   ```text
   https://your-domain.example/?token=<HUMAN_REVIEW_TOKEN>
   ```
